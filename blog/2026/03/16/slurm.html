Official ACP SLURM Restructuring & 2026 System Updates

March 16, 2026 · 3 min read
Montage Team
High Performance Computing Team

The SLURM restructuring for the ACP is officially complete. We have transitioned away from complex partition naming to a simplified model designed for maximum efficiency and fairness. These updates ensure that all researchers have equitable access to our high-performance resources.

1. Simplified Partitions​

We have consolidated all compute nodes into two primary partitions. If your .sbatch scripts still use legacy names, your jobs will not start. Please update your scripts to use:

  • cpu: General-purpose CPU computing nodes.
  • gpu: High-performance GPU-accelerated nodes.

2. Training and Dashboard Demo​

To help everyone adapt to these changes, we urge all users to join our upcoming training session. We will be demoing the new MONTERA All-in-1 dashboard, which will soon feature real-time CPU% and Memory% usage tracking.

Sign up for Training: https://forms.gle/HLreQ2kXnTgFUPAr9

Monitoring your job's efficiency is critical. If your code shows 0% CPU utilization, it is a sign of an error or a hang, and the job should be terminated immediately to avoid resource hogging.

3. The Priority Game (QoS)​

Your job's priority and duration are now defined by the Quality of Service (QoS) flag:

  • short: 2-hour limit. Use this for high-priority tests and debugging.
  • normal: The default setting with a 7-day partition limit.
  • long: For 28-day marathons. These have lower priority but are protected from preemption.

4. Memory & Fairness Model​

Our new fairness model rewards users who request smaller resource footprints. Requesting excessive RAM relative to CPU cores will lower your fairshare priority and increase your wait time.

  • Default Allocation: 4GB per CPU.
  • Maximum Recommended: 16GB per CPU.

5. CPU Spillover and Efficiency​

To ensure the cluster remains efficient, we have implemented a spillover policy for CPU jobs:

  • Jobs <= 1 Day: can execute on both CPU nodes (mum-hpc2-comp[1-4]) and idle GPU nodes (mum-hpc2-gpu[1-4]).
  • Jobs > 1 Day: remain strictly on CPU nodes to protect GPU resources for researchers who require them.

6. High-Performance GPU Hardware​

The gpu partition has been updated with high-end hardware for public use:

  • A100 GPUs: Available on nodes gpu5 and gpu6.
  • H100 GPUs: Available on nodes gpu7 and gpu8.

These specific nodes have no Multi-Instance GPU (MIG) configured, providing full access to the hardware for parallel GPU training. To request these, use --gpus=a100:1 or --gpus=h100:1 in your scripts.

7. Script Checkout​

Ensure your Slurm scripts reflect the new rules to avoid queuing delays:

#SBATCH --partition=cpu           # Options: cpu, gpu 
#SBATCH --qos=short               # High-priority tests (2hr limit)
#SBATCH --cpus-per-task=1         # Standard allocation
#SBATCH --mem=4G                  # Fairshare-friendly memory
#SBATCH --time=2:00:00            # Be accurate to start sooner

The ACP is currently tuned for maximum throughput. Use the show_cluster command to check real-time availability. If you are unclear about any of these rules, we will be available during the training session to guide you.
